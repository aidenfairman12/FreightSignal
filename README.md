# FreightSignal

**Supply chain disruption intelligence via retrieval-augmented generation.**

FreightSignal monitors live logistics and trade news and answers natural language questions about supply chain disruptions — citing the exact article chunks it used to generate each answer. Built with open-source models and evaluated rigorously with RAGAS.

---

## What it does

Ask: *"What's disrupting US port operations right now?"*

FreightSignal retrieves the most relevant chunks from its live article corpus, passes them to Llama 3.3 70B, and returns a grounded answer with full source attribution — so you can verify every claim.

The **How it works** page shows the live RAGAS scorecard (faithfulness, context precision, recall, answer relevancy) so the model's quality is always transparent.

---

## Architecture

```
RSS feeds (FreightWaves, Supply Chain Dive, JOC, DC Velocity, Reuters)
        ↓  fetch_articles.py  [GitHub Actions, every 6h]
  articles.jsonl  (raw article store)
        ↓  chunk_embed.py  [BAAI/bge-small-en-v1.5]
  Chroma vector store  (cosine similarity index)
        ↓  query-time: ANN retrieval → cross-encoder rerank → top-5 chunks
  Groq Llama 3.3 70B  →  grounded answer + source metadata
        ↓
  FastAPI  →  Next.js frontend  (Vercel)
```

**Retrieve-then-rerank**: Chroma returns 20 ANN candidates; `BAAI/bge-reranker-base` (cross-encoder) rescores each `(query, chunk)` pair and keeps the top 5. Cross-encoders are significantly more accurate than cosine similarity at the cost of latency — standard practice since Nogueira & Cho (2019).

## Tech stack

| Layer | Technology |
|-------|-----------|
| Embeddings | BAAI/bge-small-en-v1.5 (sentence-transformers) |
| Reranker | BAAI/bge-reranker-base (cross-encoder) |
| Vector store | Chroma (SQLite-backed, committed to repo) |
| Generation | Llama 3.3 70B via Groq API |
| Evaluation | RAGAS (Es et al., arXiv:2309.15217) |
| Backend | FastAPI (Python 3.11) |
| Frontend | Next.js 14 (App Router, TypeScript, Tailwind) |
| Data refresh | GitHub Actions (cron, every 6h) |
| Hosting | Render (API) + Vercel (frontend) |

All models are open-source. No OpenAI dependency.

## Running locally

### Prerequisites
- Python 3.11+
- Node.js 18+
- Free [Groq API key](https://console.groq.com)

### 1. Configure environment
```bash
cp .env.example .env
# Add your GROQ_API_KEY
```

### 2. Build the corpus and vector store
```bash
cd pipeline
pip install -r requirements.txt

# Fetch articles from RSS feeds
python fetch_articles.py

# Chunk + embed → Chroma
python chunk_embed.py
```

### 3. Start the API
```bash
cd api
pip install -r requirements.txt
uvicorn main:app --reload
# → http://localhost:8000
# → http://localhost:8000/docs
```

### 4. Start the frontend
```bash
cd frontend
npm install
# Create frontend/.env.local:
echo "NEXT_PUBLIC_API_URL=http://localhost:8000" > .env.local
npm run dev
# → http://localhost:3000
```

### 5. (Optional) Generate RAGAS evaluation scores
```bash
cd pipeline
python generate_eval_set.py   # generates synthetic QA pairs
python run_ragas_eval.py      # scores the pipeline, writes data/eval/ragas_results.json
```
Scores are served by `GET /scorecard` and displayed in the frontend.

## Automatic corpus refresh

A GitHub Actions workflow (`.github/workflows/refresh.yml`) fetches new articles and rebuilds the vector store every 6 hours. If the corpus changes, it commits the updated `articles.jsonl` and `chroma_db/` files and pushes — which triggers a Vercel redeploy automatically.

No manual intervention needed after the initial setup.

## Deployment

### API → Render (free tier)
1. Create a new **Web Service** on [render.com](https://render.com)
2. Root directory: `api/`
3. Build command: `pip install -r requirements.txt`
4. Start command: `uvicorn main:app --host 0.0.0.0 --port $PORT`
5. Add `GROQ_API_KEY` as an environment variable

### Frontend → Vercel
1. Import the repo on [vercel.com](https://vercel.com)
2. Framework preset: Next.js
3. Root directory: `frontend/`
4. Add `NEXT_PUBLIC_API_URL` pointing to your Render service URL

## Evaluation methodology

RAGAS evaluates the pipeline against a synthetic test set generated by prompting the LLM to produce realistic questions from held-out articles, then scoring the full RAG pipeline on each:

| Metric | What it measures |
|--------|----------------|
| **Faithfulness** | Does the answer contain only claims supported by the retrieved sources? (hallucination rate) |
| **Answer Relevancy** | Does the answer address the question? |
| **Context Precision** | Are the retrieved chunks actually relevant to the question? |
| **Context Recall** | Do the retrieved chunks cover the information needed to answer correctly? |

Reference: Es et al., *RAGAS: Automated Evaluation of Retrieval Augmented Generation*, arXiv:2309.15217 (2023).
