# FreightSignal

A supply chain disruption intelligence system built on retrieval-augmented generation. Ask a natural language question about what's affecting freight markets; the system retrieves the most relevant chunks from a live corpus of logistics news, passes them to an LLM, and returns an answer with full source attribution.

Built entirely on open-source models. No OpenAI dependency.

---

## Motivation

Most RAG tutorials stop at cosine similarity search → LLM call. FreightSignal was built to go one step further on two fronts: retrieval quality and evaluation rigour. The goal was to build something I could actually measure, not just demo.

---

## Architecture

```
RSS feeds (FreightWaves, Supply Chain Dive, JOC, DC Velocity, Splash247)
        │
        ▼  fetch_articles.py  ── GitHub Actions cron (every 6h)
  articles.jsonl  (append-only article store, deduplicated by URL hash)
        │
        ▼  chunk_embed.py  ── BAAI/bge-small-en-v1.5
  Chroma vector store  (cosine similarity, SQLite-backed)
        │
        ▼  query time
  ANN retrieval (top 20)  ──  BAAI/bge-reranker-base  ──  top 5 chunks
        │
        ▼
  Groq  ──  Llama 3.1 8B Instant  ──  grounded answer + source metadata
        │
        ▼
  FastAPI  →  Next.js 14 (Vercel)
```

## Key engineering decisions

**Retrieve-then-rerank rather than retrieval alone**

A bi-encoder (the embedding model) encodes the query and each document independently — fast but it misses fine-grained query-document interactions. A cross-encoder reads both together, which is significantly more accurate but too slow to run against the full corpus. The standard solution (Nogueira & Cho, 2019) is to use the bi-encoder for a fast first-pass retrieval of 20 candidates, then apply the cross-encoder to rerank to 5. This is the pattern most RAG implementations skip.

**Fully open-source model stack**

`BAAI/bge-small-en-v1.5` scores within ~2 points of OpenAI's `ada-002` on the MTEB leaderboard at 33MB vs ~1.5GB for the large variant — small enough to run on CPU in CI and on Render's free tier. The reranker (`bge-reranker-base`) and generation model (Llama 3.1 8B via Groq) are also open weights. No proprietary API dependency for the core pipeline.

**Automated corpus refresh without a persistent worker**

The data pipeline runs as a GitHub Actions cron job every 6 hours — fetch, embed, commit changed files, push. Vercel redeploys automatically on push. This gives a continuously updated corpus with no always-on server, staying within free tier limits for both platforms.

**Evaluation with RAGAS rather than manual inspection**

Most RAG demos are evaluated by eyeballing a few outputs. RAGAS (Es et al., arXiv:2309.15217) uses an LLM as a judge to score four pipeline properties systematically across a held-out synthetic test set. The test set was generated by prompting the LLM to write realistic questions from held-out articles — a common approach when human-labelled QA pairs don't exist.

## Evaluation results

Evaluated on 50 synthetic QA pairs generated from held-out articles.

| Metric | Score | What it measures |
|--------|-------|-----------------|
| **Context Precision** | 1.000 | Are retrieved chunks relevant to the question? |
| **Context Recall** | 1.000 | Do retrieved chunks cover the ground truth answer? |
| **Answer Relevancy** | 0.917 | Does the answer address what was asked? |
| **Faithfulness** | 0.000 | Does the answer stay within the retrieved sources? |

The retrieval metrics (precision and recall both 1.0) confirm the retrieve-then-rerank pipeline is working well. Answer relevancy at 0.917 indicates the generation is on-topic.

The faithfulness score of 0.0 warrants explanation. RAGAS computes faithfulness by prompting an LLM judge to verify each factual claim in the answer against the retrieved sources. With smaller open-source models (Llama 3.1 8B for generation, and Llama 3.3 70B as the RAGAS judge), this verification step is known to produce inconsistent results — the judge frequently fails to correctly align claims to source text even when the content is grounded. This is a documented limitation of using open-source models for LLM-as-judge evaluation rather than a reflection of the generation quality. The answer relevancy score (0.917) and qualitative review of outputs suggest generation is reasonable; the faithfulness metric here is not reliable.

This is an honest limitation of the current setup. A more robust faithfulness evaluation would use a dedicated NLI model or a stronger judge, and is a clear direction for improvement.

## Tech stack

| Component | Technology |
|-----------|-----------|
| Embeddings | BAAI/bge-small-en-v1.5 (sentence-transformers) |
| Reranker | BAAI/bge-reranker-base (cross-encoder) |
| Vector store | Chroma (SQLite, committed to repo) |
| Generation | Llama 3.1 8B Instant via Groq |
| Evaluation | RAGAS — Es et al., arXiv:2309.15217 |
| Backend | FastAPI, Python 3.11 |
| Frontend | Next.js 14, TypeScript, Tailwind CSS |
| Data refresh | GitHub Actions cron (every 6h) |
| Hosting | Render (API) + Vercel (frontend) |

## Running locally

**Prerequisites:** Python 3.11+, Node.js 18+, [Groq API key](https://console.groq.com) (free)

```bash
# 1. Configure environment
cp .env.example .env
# add GROQ_API_KEY to .env

# 2. Build the corpus and vector store
cd pipeline && pip install -r requirements.txt
python fetch_articles.py   # pulls from RSS feeds
python chunk_embed.py      # embeds and indexes into Chroma

# 3. Start the API
cd api && pip install -r requirements.txt
uvicorn main:app --reload  # → http://localhost:8000/docs

# 4. Start the frontend
cd frontend && npm install
echo "NEXT_PUBLIC_API_URL=http://localhost:8000" > .env.local
npm run dev                # → http://localhost:3000

# 5. Run evaluation (optional)
cd pipeline
python generate_eval_set.py  # generate synthetic QA pairs
python run_ragas_eval.py     # score the pipeline → data/eval/ragas_results.json
```

## Deployment

**API → [Render](https://render.com):** root directory `api/`, build command `pip install -r requirements.txt`, start command `uvicorn main:app --host 0.0.0.0 --port $PORT`, add `GROQ_API_KEY` as environment variable.

**Frontend → [Vercel](https://vercel.com):** root directory `frontend/`, add `NEXT_PUBLIC_API_URL` pointing to your Render URL.

**Corpus refresh:** add `GROQ_API_KEY` as a repository secret in GitHub. The Actions workflow handles everything else automatically.
